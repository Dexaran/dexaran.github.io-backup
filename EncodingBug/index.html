For those who need explanation of the described bug with encoding data of transactions...
This is a transaction simulator. It has no checks for input values. You can try it yourself.
You must input a valid address and a valid UINT256 number and click WRITE. It will encode your data using standard SolidityCoder.encodeData. Than it will DECODE encoded data and alert a decoded result.
You can now delete some characters from address, click WRITE again and you will see zeros at the left part appears. It caused by address encoding. I can't realize how you can affect UINT256 number inputing a wrong address with standard SolidityCoder. Only if someone will use custom coder on their server...
P.S. Yes. If there is no address validity check you can throw your tokens/Ether to a shorter address filled with zeros.
